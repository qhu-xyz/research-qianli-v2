### Code Findings
1. **HIGH — Threshold optimization/deployment mismatch can zero out positives.** `find_optimal_threshold()` optimizes using `precision_recall_curve` thresholds (`ml/threshold.py:33-46`), while predictions are applied with strict `>` (`ml/threshold.py:64`, used in `ml/pipeline.py:120`). PR curve semantics are threshold-inclusive, so using strict `>` can make realized predictions strictly more conservative than the optimized point. This is consistent with `threshold=0.820287` and `pred_binding_rate=0.0` despite high reported `max_fbeta` in metadata (`registry/v0001/meta.json:7`).
2. **HIGH — Threshold is tuned and evaluated on the same validation split (selection leakage).** The pipeline finds threshold on `y_val, val_proba` (`ml/pipeline.py:104-110`) and then reports final gates on that same split (`ml/pipeline.py:123-125`). This gives optimistic estimates for threshold-dependent metrics (`S1-REC`, `S1-CAP@K`, precision/F1) and weakens inferential validity.
3. **MEDIUM — Markdown pass column can disagree with JSON pass logic on missing metrics.** In `build_comparison_table`, `value is None` only prints `"--"` and does not fail the group (`ml/compare.py:169-170`), while canonical logic in `evaluate_overall_pass` marks `passed is not True` as fail (`ml/compare.py:127-135`). If a Group A metric is missing, table may show pass when it should not.
4. **MEDIUM — Memory scaling risk for real-data mode.** Features are materialized as dense NumPy arrays (`ml/features.py:36`) while original Polars frames remain live through later phases (`ml/pipeline.py:77-91`), causing multiple large in-memory copies for large datasets.
5. **LOW — Technical debt: duplicated/unused control paths.** `scale_pos_weight_auto` is configurable (`ml/config.py:95`, serialized at `ml/pipeline.py:147`) but ignored in training (`ml/train.py:40-54` always auto-computes). Pass aggregation logic is also duplicated between `build_comparison_table` and `evaluate_overall_pass` (`ml/compare.py:161-194` vs `ml/compare.py:125-135`), increasing drift risk.

### Recommendations
1. Align threshold semantics end-to-end: either change `apply_threshold` to `>=` or adjust chosen threshold (e.g., subtract epsilon) so realized classification matches PR-curve optimization semantics.
2. Remove threshold leakage by introducing a dedicated threshold-tuning split (or nested CV), then reserve validation/test for final reporting only.
3. Refactor `build_comparison_table` to consume `evaluate_overall_pass(gate_results)` directly; treat `value is None` as fail for Group A unless explicitly exempted.
4. For large datasets, reduce memory pressure by streaming/chunking feature generation or releasing DataFrames before model fitting and keeping only necessary arrays.
5. Wire `scale_pos_weight_auto` into `train_classifier` behavior or remove it from config until implemented.

### Gate Calibration
No floor adjustment is justified from Iteration 1 because all deltas vs `v0` are exactly zero and sample size is too small for stable inference (`n=20`, `n_positive=2`). Keep floors unchanged for now, but after first real-data run:
- replace global `noise_tolerance=0.02` with metric-specific uncertainty-aware tolerances (or CIs),
- make `S1-VCAP@K`/`S1-CAP@K` informational when `K >= n`,
- revisit whether Group B should remain non-blocking if `S1-REC` is mission-critical.
