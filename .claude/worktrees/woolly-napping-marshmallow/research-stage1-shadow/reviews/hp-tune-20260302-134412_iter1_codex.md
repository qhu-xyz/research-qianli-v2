Both benchmark and single-run pipeline optimize threshold on validation data and then evaluate on the same validation rows, which inflates REC/CAP/precision/Brier estimates. ### Recommendations
1. Do not promote v0003 on performance merit; it is a slight regression on the precision-critical ranking stack (AUC/AP/NDCG).
2. Keep the search centered near v0 defaults for next HP iteration (smaller local steps), and require month-level consistency checks (for example: at least 8/12 non-negative deltas on AUC and AP) before accepting mean gains.
3. Fix threshold methodology before trusting Group B movement: use a three-way split (fit / threshold-tune / final eval) or cross-fit thresholding.
4. Align threshold application with tuning semantics (`>=`) and rebaseline REC/CAP metrics after this fix.

### Gate Calibration
Current calibration is permissive relative to observed variance:
- `noise_tolerance=0.02` is very loose for stable metrics like AUC (monthly std ~0.015), making Layer 3 weakly discriminative.
- Tail floors are not currently binding (0 failures everywhere for Group A), so Layer 2 is providing little practical protection.
- S1-VCAP@100 floor/tail floor are negative, which makes this gate rarely binding despite being in Group A.

Suggested calibration direction after a few more real-data runs:
1. Move from a single global tolerance to metric-specific tolerance (tighter for AUC/AP/NDCG, looser for VCAP@100).
2. Tighten tail offsets for Group A to make Layer 2 informative.
3. Require an explicit champion reference (or fixed baseline like v0) so Layer 3 always enforces non-regression.
