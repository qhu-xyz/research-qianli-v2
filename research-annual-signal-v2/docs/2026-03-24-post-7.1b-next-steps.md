## Immediate decision

### Option A: Publish `7.1b` now

Choose this if the goal is to lock the round-sensitive `v0c` release first.

Benefits:

- freezes the current release candidate while the code path is still simple
- avoids mixing release work with structural migration
- gives a stable baseline before any further model experiments

### Option B: Defer publish briefly and do one final release review

Choose this if you want one explicit final sign-off pass on:

- manifest
- smoke test
- comparison artifact
- docs

This should be a short review, not a new implementation cycle.

One concrete check in that final review:

- rerun the publish smoke test once after the `constraint_limit` output-schema fix

Recommendation:

- publish `7.1b` first unless there is a specific unresolved release concern

## What comes next after `7.1b`

## 1. Lowest-risk repo migration batch

This is the first structural step after the release decision.

Goal:

- start moving code into the new package layout without breaking imports

Move first:

- `ml/bridge.py` -> `ml/markets/miso/bridge.py`
- `ml/realized_da.py` -> `ml/markets/miso/realized_da.py`
- split `ml/config.py` into:
  - `ml/core/calendars.py`
  - `ml/markets/miso/config.py`

Migration rule:

- use compatibility shims
- do not do a large atomic move

Acceptance:

- old imports still work
- tests still pass
- active production entrypoints still run unchanged

Why this batch first:

- lowest caller count
- lowest risk to release-critical paths
- establishes the MISO/core boundary cleanly

## What not to mix together

Do not combine these into one batch:

- publishing `7.1b`
- large code relocation
- specialist-model productionization
- `Bucket_6_20` round-aware rebuild

Keep them separate:

- release freeze
- migration batch
- model experiments

This lowers risk and keeps cause/effect clear.
