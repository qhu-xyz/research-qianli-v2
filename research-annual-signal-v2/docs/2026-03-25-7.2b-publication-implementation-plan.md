# 7.2b Publication Implementation Plan

**Date**: 2026-03-25  
**Purpose**: implementation handoff for the production `7.2b` publication path  
**Status**: review draft

## 1. Goal

`7.2b` has two distinct layers:

1. **Branch-level scorer parity**
   - match the research candidate at the branch scorer layer
   - base model
   - specialist model
   - cumulative reserved branch sets at `K=200` and `K=400`

2. **Constraint-level publication policy**
   - publish constraints, not branches
   - enforce per-tier quota at the **constraint** level after expansion
   - each 200-constraint tier must target:
     - `150` base-origin constraints
     - `50` specialist-origin constraints

This means the production publish path is:

- research-parity branch scorers
- production-specific constraint-tier assembly

## 3. What Materially Needs To Be Built

The missing work is not only “turn branch into constraint.”

### 3.1 Publish-facing adapter over the existing scorer path

Need a thin publish-facing adapter over the existing `7.2b` branch scorer code that:

- trains base model on historical cells
- trains specialist on historical true-dormant cells
- scores the target publish cell ex ante
- returns branch-level scores and provenance inputs needed by the publisher

This is **not** a new model recipe.
It is a publish-facing wrapper over the existing branch-level `7.2b` training/scoring code.

Required branch-level outputs:

- `branch_name`
- `base_score`
- `specialist_score`
- `is_true_dormant`
- `origin`
- `shadow_price_da`
- `da_rank_value`

### 3.2 Provenance assignment before expansion

Every branch must be tagged before expansion:

- `origin = "specialist"` if the branch is selected into the specialist reserved pool
- `origin = "base"` otherwise

Every constraint expanded from that branch inherits the same origin.

### 3.3 Constraint-level two-pool publication walk

This is the main new logic.

After branch -> constraint expansion:

- split constraints into:
  - `base_pool`
  - `specialist_pool`

Pool sort order must be deterministic:

- `base_pool`
  - `base_score` descending
  - `constraint_id` ascending
- `specialist_pool`
  - `specialist_score` descending
  - `base_score` descending
  - `constraint_id` ascending

Note:

- multiple constraints expanded from the same branch share the same
  `specialist_score` and `base_score`
- `constraint_id` ascending is therefore the within-branch tie-break

Tier assembly contract:

- `5` tiers
- `200` constraints per tier
- per tier:
  - take next-best `50` specialist-origin constraints
  - take next-best `150` base-origin constraints
- if specialist underfills:
  - return shortfall to base **within the same tier**
- tier is locked before moving to the next tier

### 3.4 Branch cap and SF dedup must stay online and shared within tier

Branch cap and grouped SF dedup are **not** preprocessing.

They must be enforced during the same sequential walk that fills each tier.

Implementation detail:

- each tier has two sub-walks:
  - specialist-origin fill first
  - base-origin fill second
- branch-cap state and SF-dedup state must be shared across both sub-walks inside
  the same tier
- a specialist-origin constraint selected earlier in the tier must be able to block a
  similar base-origin constraint later in that same tier

Implication:

- a constraint rejected in tier 3 could have been accepted in tier 0
- therefore pre-deduping pools before the walk would be wrong

### 3.5 Replace v0c-only publisher ordering

Current publisher behavior is:

- compute `v0c_score`
- sort one global constraint stream by `v0c_score`
- fill tiers from that single stream

`7.2b` publish behavior must instead:

- use branch scores from the new base/specialist scorer
- build two constraint pools
- fill tiers using the quota walk
- assign final rank from tier-local publication order

### 3.6 Publish-path integration

Use a separate `publish_signal_72b(...)`.

Requirements:

- coexist cleanly with `7.1b`
- preserve current `7.1b` behavior unchanged
- fail loudly on missing required `7.2b` inputs
- avoid forcing the structurally different `7.2b` walk into the `7.1b` function

## 5. Additional hardening required for publish mode

The current branch-level evaluation code still contains exploratory fallbacks that are
not acceptable for publication.

These must be hardened specifically for publish mode:

- no deviation-feature zero-fill fallback
  - current behavior exists in `ml/markets/miso/release_candidate.py`
- no skipped training cells
- no silent downgrade from missing required publish inputs
- no claim of successful publish if required branch-level scorer inputs were skipped

## 6. Recommended Build Order

### Phase 1: branch scorer handoff object

Build one publish-facing scorer function that returns all branch-level inputs needed for publication.

Deliverable:

- scored branch table for one `(planning_year, aq_quarter, class_type, market_round)` cell

### Phase 2: provenance-aware expansion

Expand scored branches to constraints and carry:

- `origin`
- `base_score`
- `specialist_score`

Deliverable:

- constraint candidate table with provenance

### Phase 3: dual-pool tier walk

Implement the real `7.2b` publish logic:

- `specialist_pool`
- `base_pool`
- sequential 5-tier fill
- online branch cap and SF dedup

Deliverable:

- published 1000-constraint selection with tiers

### Phase 4: publisher integration

Wire the new mode into the production entrypoint without regressing `7.1b`.

Deliverable:

- callable publish path for `7.2b`

### Phase 5: release/test/audit

Add:

- manifest
- smoke test
- unit tests
- integration test
- publication audit output

## 7. Explicit Non-Goals

The following are already solved or intentionally separate:

- re-deriving the research candidate contract from scratch
- changing the `7.1b` publish path semantics
- proving final 1000-constraint publication exactly matches research
  - it will not; publication policy is production-only

## 8. Confirmed Review Decisions

- publish integration should use a separate `publish_signal_72b()`
- within-tier display order should be:
  - specialist-origin block first
  - base-origin block second
- specialist shortfall should backfill from base only within the current tier
- no cross-tier reshuffling is allowed
- pool exhaustion can stop publication below 1000 if documented explicitly in the audit
- the manifest should explicitly record that the constraint-tier assembly is
  production-only and was not benchmarked in research
