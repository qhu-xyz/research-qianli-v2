# Phase 4b Design: Value-Aware Track B with Dynamic Allocation

**Date**: 2026-03-14
**Status**: Draft
**Depends on**: Phase 3 (two-track infra), Phase 4a (established that binary+weights is insufficient)

---

## 1. Problem Statement

Phase 4a showed that tiered weighting improves NB12_SP capture over Phase 3's unweighted
binary, but only at K=100 (0.0526 vs 0.0049). At K=50, Phase 4a scored NB12_SP=0.0000 —
worse than Phase 3. The fundamental problems are:

1. **Wrong objective**: Binary classification optimizes "did it bind," not "did it bind for
   meaningful value." A $50 binder and a $500k binder are still identical positives even with
   tiered sample weights. The model predicts P(bind), not E[SP].

2. **Forced allocation**: Fixed R=5 means 5 dormant slots are always spent, even when the model
   has no good dormant candidates. In most holdout groups, 4 of 5 slots hold non-binders with
   $0 SP. Those wasted slots degrade VC@50 for nothing.

3. **Weak features**: The 12 density bin maxima (AUC ~0.66) describe forward-looking SP
   distributions but miss the strongest predictive signals: which specific constraints does
   this branch touch, and are those constraints actively binding on other branches?

4. **Dormant is too broad**: A branch that last bound 13 months ago and one that last bound
   5 years ago are both `bf_combined_12 == 0`, but they are completely different prediction
   problems. Lumping them together dilutes the signal from recoverable recent-dormant branches.

**Target**: NB12_SP@50 ≥ 10% (currently ~1%), NB12_SP@100 ≥ 10% (currently ~5%).

## 2. Scope

Phase 4b addresses all four problems simultaneously:

1. **Value-aware target** — regression on `log1p(SP)` or two-stage `P(bind) × E[SP|bind]`
2. **Dynamic slot allocation** — score-thresholded R, unused slots return to Track A
3. **New feature groups** — constraint propagation, recency, density shape, persistence
4. **Dormant subcohorts** — recent_dormant vs deep_dormant (history_zero excluded)

Phase 4b does NOT:
- Change Track A scoring (still v0c formula)
- Attempt to solve history_zero (separate feature-engineering problem)
- Modify `ml/evaluate.py` or `ml/merge.py` beyond what's needed for dynamic R

## 3. Dormant Subcohorts

```python
# Phase 4a (current):
dormant = (bf_combined_12 == 0) & has_hist_da

# Phase 4b:
recent_dormant = (bf_combined_12 == 0) & (months_since_last_bind <= 24)
deep_dormant   = (bf_combined_12 == 0) & (months_since_last_bind > 24) & has_hist_da
history_zero   = ~has_hist_da   # excluded from Track B entirely
```

`recent_dormant` is where most recoverable NB SP lives — branches that were active last
planning year and might reactivate. `deep_dormant` needs stronger features (constraint
propagation) to be useful.

`months_since_last_bind` is computed from realized DA history: the number of months between
the binding frequency cutoff month and the most recent month where this branch had
`realized_shadow_price > 0` on any constraint.

Both subcohorts are trained together (single model) but `months_since_last_bind` acts as a
natural feature that lets the model learn different decision boundaries for each.

## 4. New Features

### 4.1 Constraint Propagation Features (highest expected impact)

These capture the "cold branch, hot constraint" signal: a dormant branch whose constraints
are actively binding on OTHER branches.

For each dormant branch, look up its constraint IDs. For each constraint ID, compute binding
frequency and mean SP on other branches in the last 12 months. Aggregate back to branch:

| Feature | Definition |
|---------|-----------|
| `max_cid_bf_other` | Max binding frequency of this branch's CIDs on OTHER branches (12mo) |
| `mean_cid_bf_other` | Mean binding frequency across CIDs on other branches |
| `sum_cid_sp_other` | Total SP generated by this branch's CIDs on other branches (12mo) |
| `max_cid_sp_other` | Max SP of any single CID on other branches |
| `n_active_cids_other` | Count of this branch's CIDs that bound on at least 1 other branch |
| `active_cid_ratio_other` | `n_active_cids_other / count_cids` |

Data source: the same realized DA / binding frequency tables already used for `bf_combined_12`.
The join is: for each (branch, cid) pair in the dormant branch, find all OTHER branches
containing that cid, look up their realized SP for the same period.

### 4.2 Recency Features

| Feature | Definition |
|---------|-----------|
| `months_since_last_bind` | Months from BF cutoff to most recent binding month (∞ if never) |
| `historical_max_sp` | Peak single-quarter SP from any period older than 12 months |
| `historical_sp_12_24m` | Total SP in months 13-24 |
| `historical_sp_24_36m` | Total SP in months 25-36 |
| `n_historical_binding_quarters` | Count of quarters where branch had SP > 0 (all history) |

### 4.3 Density Shape Features

Computed from the existing density distribution (the bin columns already loaded).

| Feature | Definition |
|---------|-----------|
| `tail_mass_ge_100` | Sum of density bins ≥ 100 (right-tail probability mass) |
| `tail_mass_ge_110` | Sum of density bins ≥ 110 |
| `density_entropy` | Shannon entropy of the density distribution |
| `density_skewness` | Skewness of the density distribution |
| `density_kurtosis` | Kurtosis of the density distribution |
| `density_cv` | Coefficient of variation of the density distribution |

### 4.4 Persistence & Aggregation Features

| Feature | Definition |
|---------|-----------|
| `n_quarters_with_signal` | Number of quarters where the branch appears in signal data |
| `cid_std` | Std dev of per-CID density max across CIDs |
| `cid_top2_mean` | Mean of the top-2 CID density maxima |
| `active_cid_ratio` | `count_active_cids / count_cids` |

### 4.5 Feature Summary

| Group | Count | Data Source | Build Effort |
|-------|:---:|---|---|
| Current Phase 4a | 12 | density bins (existing) | Done |
| Constraint propagation | 6 | binding history + cid mapping | Medium |
| Recency | 5 | realized DA history | Low |
| Density shape | 6 | density bins (derived) | Low |
| Persistence/aggregation | 4 | signal + density (derived) | Low |
| **Total** | **33** | | |

Build order: constraint propagation + recency first (highest impact, parallel), then
shape + persistence (derived from existing data, fast).

## 5. Modeling Approaches — Three Experiments

### 5.1 Experiment 1: `tb_reg_value` — Direct Value Regression

- **Population**: recent_dormant + deep_dormant (not history_zero)
- **Target**: `log1p(realized_shadow_price)` — continuous regression
- **Model**: LightGBM regression (`objective: regression`, `metric: rmse`)
- **Features**: all 33 (current 12 + new 21)
- **Slot allocation**: dynamic R (see §6)
- **Score**: predicted `log1p(SP)` directly ranks by expected value

This is the simplest modeling change: one model, one target, direct value prediction.
The 92% zeros problem is mitigated by: (a) the new features provide stronger signal for
separating binders from non-binders, and (b) the regression naturally ranks high-SP
predictions above low-SP predictions.

LightGBM params:
```python
params = {
    "objective": "regression",
    "metric": "rmse",
    "learning_rate": 0.03,
    "num_leaves": 15,
    "subsample": 0.8,
    "colsample_bytree": 0.8,
    "min_child_samples": 5,
    "num_threads": 4,
    "verbose": -1,
}
```

### 5.2 Experiment 2: `tb_two_stage_ev` — Expected Value

- **Stage 1**: P(bind) binary classifier (LightGBM, same as Phase 4a but with all 33 features)
- **Stage 2**: `log1p(SP)` regression on **predicted-positive subset** (P(bind) > 0.5)
- **Final score**: `P(bind) × expm1(predicted_log1p_sp)`
- **Slot allocation**: dynamic R on expected value

The two-stage avoids the zero-inflation problem: Stage 1 filters non-binders, Stage 2
predicts magnitude among likely binders. More complex, only worth running if Experiment 1
shows regression target helps but struggles with the zero-dominated distribution.

### 5.3 Experiment 3: `tb_uplift_on_v0c` — Uplift Model (safest)

- **Population**: dormant branches only
- **Target**: `log1p(realized_shadow_price) - v0c_rank_percentile` (residual over v0c)
- **Final score for dormant**: `v0c_score + α × uplift_score`
- **No hard track separation** — dormant branches compete directly with established
- **α** tuned on dev NB12_SP@50

This is the safest approach: no forced slot allocation, dormant branches only enter top-K
if their uplift-adjusted score beats the marginal Track A candidate. Preserves VC by
construction since you're augmenting v0c, not replacing it.

### 5.4 Experiment Order

1. **Build features first** — constraint propagation + recency are independent of modeling
   choice and benefit all 3 experiments.
2. **Run Experiment 3 (uplift) first** — safest, no forced slots, directly tests whether
   better features + value scoring can surface high-SP dormant branches without degrading VC.
3. **Run Experiment 1 (regression) second** — simpler modeling, tests whether direct
   log1p(SP) regression outperforms uplift.
4. **Run Experiment 2 (two-stage) only if Experiment 1 shows regression helps but
   zero-inflation is a problem** — the extra complexity is only justified if needed.

## 6. Dynamic Slot Allocation

### 6.1 Score-Thresholded R

Replace fixed R with:

```python
R_actual = min(R_max, count(track_b_score >= tau))
# Unused slots (R_max - R_actual) go back to Track A
```

For Experiments 1 and 2, `track_b_score` is the predicted value (log1p(SP) or expected
value). For Experiment 3, dynamic allocation is implicit — dormant branches compete on
uplift-adjusted score.

### 6.2 Threshold Calibration

`τ` is calibrated on dev to maximize NB12_SP per slot:

```python
tau = argmax over candidate thresholds:
    mean_over_dev_groups(
        captured_NB12_SP(slots where score >= tau) / count(slots where score >= tau)
    )
```

Candidate thresholds: percentiles {50, 60, 70, 80, 90} of Track B positive-class scores
on dev training data.

### 6.3 R_max Values

- K=50: R_max = 5 (same as Phase 4a)
- K=100: R_max = 15 (same as Phase 4a)

Dynamic R means actual allocation is often less: if only 2 dormant branches score above τ
at K=50, R_actual = 2 and the other 3 slots go to Track A. This directly addresses the
"4 of 5 slots wasted" problem.

### 6.4 Merge Module Change

`ml/merge.py:merge_tracks()` needs a minor extension to support dynamic R:

```python
def merge_tracks(
    track_a: pl.DataFrame,
    track_b: pl.DataFrame,
    k: int,
    r: int,
    tau: float | None = None,  # NEW: score threshold
) -> tuple[pl.DataFrame, np.ndarray]:
    # If tau is set, only include Track B branches with score >= tau
    if tau is not None:
        qualified = track_b.filter(pl.col("score") >= tau)
        r_actual = min(r, len(qualified))
    else:
        r_actual = min(r, len(track_b))
    # ... rest unchanged
```

## 7. Model Selection Scorecard

Do NOT select models on generic AUC or RMSE. Select on the actual business tradeoff:

| Metric | What it measures | Role |
|---|---|---|
| **NB12_SP per reserved slot** | Value efficiency of each dormant slot | Primary — maximize |
| **NB12_SP@50** | Total NB value capture in top-50 | Must be ≥ 5% (target 10%) |
| **NB12_SP@100** | Total NB value capture in top-100 | Must be ≥ 10% |
| **VC@50 delta vs v0c** | Cost of NB allocation | Must be > -0.01 absolute |
| **VC@100 delta vs v0c** | Cost of NB allocation at K=100 | Must be > -0.01 absolute |
| **NB12 precision in reserved slots** | Fraction of reserved slots that are actual binders | Monitor |
| **R_actual / R_max** | Slot utilization (how often τ filters out weak candidates) | Monitor |

## 8. Evaluation Design

### 8.1 Dual-K Evaluation

Same as Phase 4a: evaluate at both K=50 and K=100 with independent R allocations.
`_append_zero_history()` keeps history_zero in the evaluation universe for apples-to-apples
comparison vs v0c.

### 8.2 Dev Sweep

For each experiment:
- Build features (once, shared)
- Train on expanding-window dev splits (recent_dormant + deep_dormant only)
- Sweep τ at each K level on dev
- Report NB12_SP per slot, NB12_SP@K, VC@K, slot utilization

### 8.3 Holdout Validation

Winner from dev sweep validated on holdout. Compare against:
- v0c (no two-track) — full-universe champion
- Phase 3 (tt_v0c_r5_r15) — unweighted binary, forced R
- Phase 4a (p4a_tiered_lgbm_r5_r15) — tiered weighted binary, forced R

### 8.4 Baselines

| Version | Track B model | R allocation | Features |
|---------|---|---|---|
| v0c | None | R=0 | — |
| Phase 3 | Unweighted binary logistic | Fixed R=5/15 | 12 density bins |
| Phase 4a | Tiered weighted binary LightGBM | Fixed R=5/15 | 12 density bins |
| **Phase 4b** | Value regression / two-stage / uplift | **Dynamic R ≤ 5/15** | **33 features** |

## 9. Success Criteria

Phase 4b is successful if, on holdout at the selected configuration:

1. **NB12_SP@50 ≥ 5%** (stretch: ≥ 10%) — up from Phase 4a's 0%
2. **NB12_SP@100 ≥ 10%** — up from Phase 4a's 5.3%
3. **VC@50 delta vs v0c > -0.01 absolute**
4. **VC@100 delta vs v0c > -0.01 absolute**
5. **NB12_SP per reserved slot > Phase 4a** at matched R_max

If none of the 3 experiments reach criterion 1, NB discovery via two-track with current
data is declared infeasible with existing feature set. Next step would be constraint-level
modeling (predicting at constraint level rather than branch level).

## 11. Phase 4b Contingency

If all 3 experiments fail to reach NB12_SP@50 ≥ 5%, the next direction is
**constraint-level modeling**: predict binding at the constraint level (much more data,
stronger signal because constraints are more stable than branches), then aggregate
predicted constraint-level SP back to branches. This changes the unit of prediction
from ~800 branches to ~5000+ constraints per group.
