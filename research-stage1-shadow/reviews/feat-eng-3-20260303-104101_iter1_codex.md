### Code Findings
1. Medium: threshold selection and evaluation are done on the same validation split, which can inflate threshold-dependent metrics (S1-REC, S1-CAP@100/500) and complicate monitor-gate interpretation. Evidence in [ml/benchmark.py:75](/home/xyz/workspace/research-qianli-v2/research-stage1-shadow/ml/benchmark.py:75) and [ml/benchmark.py:81](/home/xyz/workspace/research-qianli-v2/research-stage1-shadow/ml/benchmark.py:81), also [ml/pipeline.py:105](/home/xyz/workspace/research-qianli-v2/research-stage1-shadow/ml/pipeline.py:105) and [ml/pipeline.py:123](/home/xyz/workspace/research-qianli-v2/research-stage1-shadow/ml/pipeline.py:123). This is not direct train/val leakage for ranking metrics, but it is calibration leakage for thresholded monitors.
2. Medium: multi-month gate `overall_passed` treats `None` as pass via `all(x is not False ...)`, so partially-defined gates can be marked as passing when they are actually unevaluable. See [ml/compare.py:233](/home/xyz/workspace/research-qianli-v2/research-stage1-shadow/ml/compare.py:233).
3. Low: threshold scaling is not bounded to `[0,1]`; large override factors can silently produce degenerate all-negative predictions. See [ml/threshold.py:46](/home/xyz/workspace/research-qianli-v2/research-stage1-shadow/ml/threshold.py:46) and [ml/threshold.py:64](/home/xyz/workspace/research-qianli-v2/research-stage1-shadow/ml/threshold.py:64).
4. Low (debt): source-loader feature lists are duplicated in multiple modules, increasing drift risk for future feature-engineering iterations. See [ml/config.py:16](/home/xyz/workspace/research-qianli-v2/research-stage1-shadow/ml/config.py:16), [ml/features.py:50](/home/xyz/workspace/research-qianli-v2/research-stage1-shadow/ml/features.py:50), and [ml/data_loader.py:43](/home/xyz/workspace/research-qianli-v2/research-stage1-shadow/ml/data_loader.py:43).
