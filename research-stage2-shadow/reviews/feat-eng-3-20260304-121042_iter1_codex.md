### Code Findings
1. No frozen-classifier violation in this iteration. Champion and v0011 classifier blocks are identical (features, monotone constraints, and hyperparameters), so worker changes stayed in regressor scope (`registry/v0009/config.json:2-44`, `registry/v0011/config.json:2-44`, `registry/v0011/changes_summary.md:28`).
2. Medium: potential train/test leakage ambiguity for `f0` unless upstream loader treats `train_end` as exclusive. Training window ends at `auction_month` (`ml/data_loader.py:168-173,178-183`) while test for `f0` also targets the auction month (`ml/data_loader.py:304-318`). There is no explicit assertion here that loaded training rows exclude the target test month.
3. Low: `ml/pipeline.py` docstring claims classifier overrides are supported, but `_apply_config_overrides` only applies top-level and regressor keys, silently ignoring classifier overrides (`ml/pipeline.py:333,345-351`). This is correctness debt and can mislead future experiments.
4. Low: feature-importance analysis is not materialized. `benchmark.py` expects `_feature_importance` in per-month metrics (`ml/benchmark.py:135-142`), but `evaluate_pipeline` does not emit it (`ml/evaluate.py:226-243`), so no `feature_importance.json` exists for v0011.

### Regression Quality Analysis
Binding-only calibration remains acceptable but slightly worse vs champion:
1. C-RMSE mean: 2866.6 vs 2827.4 (+39.2, +1.4%) (`registry/v0011/metrics.json:265`, `registry/v0009/metrics.json:265`).
2. C-MAE mean: 1142.5 vs 1136.7 (+5.8, +0.5%) (`registry/v0011/metrics.json:266`, `registry/v0009/metrics.json:266`).
3. Tail calibration remains harsh months-driven: worst C-RMSE 5786.2 and C-MAE 2220.2 (`registry/v0011/metrics.json:322-323`), indicating persistent stress months despite average pass.
4. Spearman mean improves slightly to 0.3925 (+0.0015), but bottom-2 Spearman is slightly worse than champion (0.2678 vs 0.2705), still within 0.02 tolerance (`registry/v0011/metrics.json:264,340`; `registry/v0009/metrics.json:264,340`).
5. Regressor-specific feature contribution cannot be quantitatively verified from artifacts because feature-importance output is missing (see finding #4).

### Recommendations
1. Accept v0011 as gate-passable, but treat it as a precision-focused tradeoff: better top-100 capture, weaker breadth (500/1000).
2. In next iteration, prioritize recovering EV-VC@500/EV-VC@1000 while preserving EV-VC@100 gains. Unified-regressor mode is a valid experiment candidate because current gated behavior appears to over-concentrate top-K value (`registry/v0011/config.json:126`).
3. Add an explicit leakage guard in data loading: assert no overlap between fit/val auction months and target test month for each run.
4. Implement feature-importance export from regressor training/evaluation so feature-engineering decisions are evidence-backed.
5. Keep classifier frozen exactly as-is; do not alter `ClassifierConfig`, classifier features, or threshold policy.

### Gate Calibration
1. Do not tighten floors yet. EV-VC@500 and EV-VC@1000 are already close to L3 failure margins.
2. Replace fixed `noise_tolerance=0.02` with a scale-aware rule. Current tolerance is too lenient for tiny-scale metrics (EV-VC@100 bottom-2 is 0.0111, so L3 is effectively non-binding) and too strict to be meaningful for large-scale error metrics without normalization (`registry/gates.json:4`; `registry/v0011/metrics.json:337,341`; `memory/hot/gate_calibration.md:25-30`).
3. Raise EV-VC@100 tail floor from near-zero over time; current tail floor (0.000135) is effectively non-protective (`registry/gates.json:40-45`).
