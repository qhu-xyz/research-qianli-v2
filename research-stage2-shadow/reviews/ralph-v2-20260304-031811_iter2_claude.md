## Summary

Iteration 2 tested two hypotheses on top of the v0005 base (L2=5/mcw=25): (A) depth reduction 5→4 and (B) L1 regularization 10x (reg_alpha 0.1→1.0). Both were screened on months 2021-11 (weak Spearman) and 2022-12 (strong EV-VC). Hypothesis B was selected as the winner via the override rule (A dropped EV-VC@100 by 0.0205 on 2021-11, exceeding the 0.01 threshold).

**Critical finding: The v0006 full 12-month benchmark appears to have run with the WRONG config (reg_alpha=0.1 instead of 1.0).** Evidence: (1) The v0006 `config.json` records `reg_alpha: 0.1`, not 1.0. (2) The full benchmark per-month values are *exactly* identical to v0005 across all 12 months and all metrics (e.g., 2021-11 EV-VC@100=0.0487 and Spearman=0.2635 in both). (3) The screening run *did* show differences — Hyp B 2021-11 EV-VC@100=0.0375 vs v0005=0.0487. If the full benchmark had used reg_alpha=1.0, the 2021-11 values should have matched the screen (0.0375), not v0005 (0.0487). The conclusion in the changes_summary that "L1=1.0 had effectively zero impact" is almost certainly wrong — the full benchmark simply re-ran v0005. The code change to `ml/config.py` was made *after* the benchmark, but the benchmark itself didn't apply the override. **This iteration's full benchmark results are invalid and should be re-run.**

## Code Review Findings

### Config Change (Correct)
The diff is minimal and correct: `reg_alpha: float = 0.1` → `reg_alpha: float = 1.0` in `RegressorConfig`, with matching test assertion update. No other files were modified.

### Config.json Discrepancy (BUG)
**`registry/v0006/config.json` line 124 records `"reg_alpha": 0.1`** — the OLD default. This proves the full benchmark ran with the base v0005 config, not with the reg_alpha=1.0 override. The registry artifact should record the config *as actually used during evaluation*, not the pre-change defaults.

The likely root cause: the benchmark script snapshots the config from `RegressorConfig()` defaults BEFORE applying overrides, or the code change was committed after the benchmark was run but the benchmark script reads from the committed code (which at that point still had reg_alpha=0.1).

### Feature Count Discrepancy
The business_context.md and test file say "24 features (13 classifier + 11 additional)" but config.py's `_ALL_REGRESSOR_FEATURES` uses `_V1_CLF_FEATURES` (29 features) + 5 additional = 34 features. The test asserts 24 features, but this is using the `ClassifierConfig()` default (v0, 13 features), not v1. The business_context appears to describe a 13+11=24 feature set, while the code has 29+5=34. The changes_summary mentions "29 active features (of the 34 configured, 5 are missing from data)" — confirming 34 configured features with 5 missing at runtime. This is not a bug per se (missing features are handled gracefully), but the documentation and test assertions should reflect reality.

Key observations from valid screen data:

1. **Neither hypothesis recovered Spearman meaningfully.** Both showed < 0.002 improvement on 2021-11 (weak month). On 2022-12 (strong month), both improved Spearman by ~0.001-0.0015. This is not enough to close the 0.0008 gap on the 12-month mean.

2. **Both hypotheses degraded EV-VC@100 on 2021-11.** Hyp A was catastrophic (-42%), Hyp B was bad (-23%). This is concerning — the regularization approaches that should help Spearman are actively hurting value capture on the weakest month.

3. **Hyp A showed a clean pattern**: strong month preserved (+0.7%), weak month degraded (-42%). Depth=4 reduces model capacity in a way that specifically hurts months with complex constraint interactions.

4. **Hyp B showed a mild pattern**: strong month slightly degraded (-2.8%), weak month moderately degraded (-23%). L1=1.0 has a detectable but small effect. The "zero impact" conclusion from the full benchmark was wrong, but the actual impact is small and negative for EV-VC@100.

## Hypothesis Validation

**The core hypothesis was wrong.** The direction_iter2.md predicted that L1 regularization would improve Spearman by zeroing out noisy features. The screening shows L1=1.0 barely moved Spearman (+0.001 on strong month, +0.0 on weak month) while degrading EV-VC@100 on both months. Similarly, depth=4 barely moved Spearman while catastrophically degrading EV-VC@100 on the weak month.

**The Spearman gap (0.0008 below floor) is not addressable by regularization alone.** Three regularization axes have now been tested:
- L2 (reg_lambda=5): Spearman went from 0.3928 to 0.3920 (made it worse)
- L1 (reg_alpha=1.0): Screen shows negligible effect on Spearman
- Depth reduction (depth=4): Screen shows negligible effect on Spearman

The Spearman compression appears to be a structural property of L2=5/mcw=25 regularization, not noise or overfitting.

## Gate Calibration Assessment

**The Spearman gate is dysfunctional.** The floor (0.3928) equals v0's exact mean. With 12 eval months and std=0.071, the standard error of the mean is ~0.021. A 0.0008 difference is 0.04 SE — pure noise. The gate is blocking promotion of a version that is meaningfully better on 3 of 4 Group A metrics because of a 0.2% regression on the 4th metric that is within measurement noise.

**Recommendation (HUMAN_SYNC required):**
1. **Minimum fix**: Lower Spearman floor to 0.3800 (97% of v0 mean). This provides 1.3% headroom while still enforcing quality.
2. **Better fix**: Set all Group A floors to 90% of v0 mean: EV-VC@100=0.0621, EV-VC@500=0.1944, EV-NDCG=0.6725, Spearman=0.3535. This is consistent with the gate system's purpose (quality floor, not "beat champion on every metric").
3. **Best fix**: Use percentile-based calibration — set floor at v0's 25th percentile per-month value. This directly measures "is the new version worse than v0's typical worst months?"

The current gates prevent any meaningful exploration because every regularization change that improves EV-VC slightly degrades Spearman. This is a fundamental trade-off, not a quality problem.

## Recommendations for Iteration 3

### Priority 1: Fix the config bug and re-run v0006

The full benchmark must be re-run with reg_alpha=1.0 actually applied. This gives us valid data on whether L1 regularization helps, hurts, or is neutral across all 12 months. The screening suggests it will be approximately neutral (small negative for EV-VC@100, negligible for Spearman), but we need the full data.

### Priority 2: Explore orthogonal axes (if re-run confirms L1 is neutral)

Regularization (L2, L1, depth) is an exhausted axis for Spearman recovery. Recommended directions:

1. **Value-weighted training** (`value_weighted=True`): Weight training samples by shadow price magnitude. This directly optimizes for what matters — accurate prediction of high-value constraints. Could improve both Spearman (by focusing the model on the high-shadow-price tail that drives rank correlation) and EV-VC (by calibrating predictions where they matter). This was identified as high-potential in D3 but deferred.

2. **Learning rate reduction** (0.05→0.03 with n_estimators=600→800): More gradual learning with more trees could stabilize predictions without the leaf-weight compression that L2 causes. This is a different mechanism than L2 for reducing variance.

3. **Depth=4 + lower L2** (depth=4, reg_lambda=2.0, mcw=15): The screen showed depth=4 hurts EV-VC@100 badly on weak months but preserves Spearman. The EV-VC loss might be recoverable by loosening the L2/mcw that depth=4 makes less necessary. Depth=4 with softer regularization is unexplored.

### Priority 3: Escalate gate calibration

Two iterations of work have been blocked by a Spearman floor that is set at v0's exact mean. v0005 provides +6.5% EV-VC@100 and +5.9% EV-VC@500 — the metrics that directly drive capital allocation — and cannot be promoted. The agentic loop is now spending iterations trying to recover 0.0008 in Spearman while risking the EV-VC gains that matter. **This should be escalated to HUMAN_SYNC before iteration 3** rather than continuing to optimize around a miscalibrated gate.

### Do NOT pursue:
- Unified regressor mode: No evidence it would help Spearman. The gated mode is correct for the business objective (predicting magnitude conditional on binding).
- Feature engineering: L1=1.0 screening showed all features survive the sparsity penalty. The feature set is not the problem.
- Subsampling: Already tested in iter 1 (Hyp B, subsample=0.6/colsample=0.6) — degraded strong months by 24%.
