# Review: F1 Implementation Plan (v3)

- [2026-03-09-f1-implementation.md](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md)

## Findings

1. High: The plan does not define a minimum-history policy for early `f1` months, so the first dev months will not actually have the intended 8-row training window. The plan says training should use "8 usable rows" [docs/plans/2026-03-09-f1-implementation.md#L25](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L25), and `collect_usable_months()` just walks backward until it finds up to 8 rows [docs/plans/2026-03-09-f1-implementation.md#L375](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L375). But the plan also expects all 30 dev months to run [docs/plans/2026-03-09-f1-implementation.md#L811](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L811). For early eval months like `2020-06`, the repo only has `f1` data starting in `2020-01`, so there are fewer than 8 usable prior `f1` rows. The plan needs an explicit rule: either skip months with `< 8` usable rows, or allow shorter history and record that the window depth changed.

2. Medium: The plan is internally inconsistent about `lag`; after Task 5, `lag` is effectively dead, but Task 7 still exposes it as a key CLI/control concept. Task 4 says row inclusion is still controlled by the training-window `lag` [docs/plans/2026-03-09-f1-implementation.md#L315](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L315), yet Task 5 replaces that logic with `collect_usable_months(m, period_type, n_months=8)` which has no `lag` parameter [docs/plans/2026-03-09-f1-implementation.md#L420](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L420). Then Task 7 still adds `--lag` and auto-computes `period_offset(period_type) + 1` [docs/plans/2026-03-09-f1-implementation.md#L559](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L559). That leaves two bad outcomes: either `--lag` becomes unused and misleading, or someone tries to re-thread it back into selection logic and reintroduces the earlier confusion. The plan should either remove `--lag` from the `f1` path or define exactly what remains controlled by it.

3. Medium: The `v1` blend-search dev result will be in-sample and optimistic, but the plan currently presents it like a normal baseline. Task 12 tunes the blend weights on the same 30 dev months that it then reports as `v1` performance [docs/plans/2026-03-09-f1-implementation.md#L851](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L851). That is fine for model selection, but not for a like-for-like dev comparison against `v0`. The plan should explicitly label `v1` dev metrics as "selected on dev" or add a nested tuning/split if it wants dev numbers to be comparable.

4. Medium: Task 6 still overstates what timestamp checks can prove about feature leakage. File mtimes for paired `f0`/`f1` snapshots [docs/plans/2026-03-09-f1-implementation.md#L496](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L496) can support a consistency check, but they do not prove the files were generated before the actual auction cutoff or that upstream feature construction avoided post-cutoff data. The line claiming this would "confirm" a pre-auction snapshot [docs/plans/2026-03-09-f1-implementation.md#L532](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L532) is too strong. This should remain an upstream assumption unless the plan adds stronger provenance evidence.

5. Low: Task 1 adds generic schedule/helpers that imply broader period-type support than the helper semantics actually provide. `MISO_AUCTION_SCHEDULE` includes `q2/q3/q4` [docs/plans/2026-03-09-f1-implementation.md#L54](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L54), while `period_offset()` just parses the trailing digit [docs/plans/2026-03-09-f1-implementation.md#L70](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L70) and `delivery_month()` applies a simple month offset [docs/plans/2026-03-09-f1-implementation.md#L75](/home/xyz/workspace/research-qianli-v2/research-stage5-tier/docs/plans/2026-03-09-f1-implementation.md#L75). That is fine for `f1`, but misleading for quarterlies, where "q4" is not just "auction + 4 months." Since these helpers are going into shared config, the plan should either keep them explicitly monthly-only or avoid introducing quarterly entries here.

## Open Questions

- Should the earliest `f1` dev months be skipped until 8 usable rows exist, or should the runner allow shorter windows and record actual depth?
- Is `--lag` meant to survive only as metadata/version naming, or should it be removed once row selection switches to `collect_usable_months()`?
- Do you want `v1` to be treated as a tuning artifact for `v2`, or as a standalone baseline whose dev metrics need a fairer evaluation protocol?

## Bottom Line

The temporal logic is mostly right now, but the plan is still not fully implementation-safe. The biggest missing decision is what to do when the repo cannot supply 8 usable prior `f1` rows; until that is specified, the plan’s expected 30-month ladder is not well-defined.
