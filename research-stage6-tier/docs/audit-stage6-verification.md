# Stage 6 Verification Audit Report

Date: 2026-03-09 (revised after review)

## Status: Incomplete Prototype

Stage 6 is a partial implementation: it rebuilds only the `binding_freq_*` feature
family with partial-month awareness. It is NOT a complete point-in-time correction
of the full feature row. The daily cache and feature reconstruction are verified
against real data, but no end-to-end experiment has been run.

---

## Scope of What Stage 6 Actually Does

### What IS rebuilt with partial-month awareness
- `binding_freq_1`, `binding_freq_3`, `binding_freq_6`, `binding_freq_12`, `binding_freq_15`
- These use `ml/lookback_history.py` which truncates month M-1 to `look_back_days`
  via the daily cache, and uses full-month cache for all older months

### What is NOT rebuilt — loaded from month-M snapshot as-is
- `v7_formula_score` (derived from `da_rank_value` + `density_ori_rank_value`)
- `da_rank_value` (historical 60-month lookback from V6.2B signal pipeline)
- `prob_exceed_110`, `constraint_limit` (Spice6 forward-looking model outputs)

The non-rebuilt features are loaded via `load_v62b_month(m, "f0", "onpeak")` at
`run_v10e_lookback_days.py:111`, which reads the V6.2B signal parquet for that
auction month. These features are generated by the signal pipeline before the
auction, so they are believed to be pre-cutoff safe for f0. However, this
assumption has not been independently verified in this audit.

`lookback_history.py:7` explicitly states the limited scope:
> "For the first implementation, only the `binding_freq_*` family is rebuilt."

## 3. Daily Realized-DA Cache Build

### Schema
```
data/realized_da_daily/{YYYY-MM}.parquet
  market_date: Date
  constraint_id: String
  shadow_price_net_day: Float64  # signed daily sum (preserves netting)
  realized_sp_day: Float64       # abs(daily sum)
```

### Why signed netting matters
Monthly semantics: `realized_sp_month = abs(sum(shadow_price across all hours))`.
Storing only `abs(daily)` would lose cross-day cancellation and break the 31-day
equivalence check.

### Execution
```bash
python scripts/cache_realized_da_daily.py
```
- Source: `MisoApTools().tools.get_da_shadow_by_peaktype()` via Ray
- Range: 2019-06 to 2025-12 (**79 months**, both daily and monthly caches)
- All 79 months cached successfully

### Reproducibility issue (Finding #4)
`cache_realized_da_daily.py` has no CLI arguments — it hardcodes the Ray address
and month range (2019-06 to 2025-12). `stage6-design.md:62` documents a `--start`
/ `--end` interface that does not exist. The ad-hoc verification checks (12-day
subset, recovery rate) were run inline in the Python REPL during this session and
have no corresponding script in the repo.

## 5. Known Issues from Review

### Finding 1 (High): Scope limited to binding_freq_* only
Stage 6 rebuilds only the binding_freq feature family. Other features
(`v7_formula_score`, `da_rank_value`, `prob_exceed_110`, `constraint_limit`) are
loaded from V6.2B/Spice6 signal parquets at the month-M snapshot. The assumption
that these are pre-cutoff safe for f0 has not been independently verified.

**Status**: By design (see `lookback_history.py:7`), but the stage6-design.md
overstates scope if read as "full point-in-time correction."

### Finding 2 (High): No end-to-end results
No stage6 result artifact exists under `registry/` or `holdout/` beyond the
stage5 controls (v0, v10e-lag1). The dev run, holdout run, v10e-lag1 comparison,
and concrete month inspection have not been executed.

**Status**: Blocked on this session. Next step is to run the experiment.

### Finding 3 (Medium): Fragile row boundary enforcement
The safe row boundary (train on M-9..M-2, not M-8..M-1) is enforced only by a
script-level workaround in `run_v10e_lookback_days.py:93-94`:

```python
shifted_eval = (eval_ts - pd.DateOffset(months=1)).strftime("%Y-%m")
train_df, _, _ = load_train_val_test(shifted_eval, ...)
```

`data_loader.py:109` still documents and implements M-8..M-1 as its default
contract. Any new script calling `load_train_val_test()` directly would get the
leaky window.

**Status**: Known tech debt. Should be fixed by adding a `production_lag` parameter
to `load_train_val_test()` or creating a safe wrapper.

### Finding 4 (Medium): Verification not fully reproducible
- `cache_realized_da_daily.py` has no CLI (hardcoded range/Ray address)
- `stage6-design.md:62` documents `--start`/`--end` flags that don't exist
- Tests 2 and 3 (12-day subset, recovery rate) were REPL-only with no script

**Status**: Should add CLI to cache script and create a verification script for
the 12-day checks.

### Finding 5 (Low): File count discrepancy
The original audit said "78 months" but both `data/realized_da_daily/` and
`data/realized_da/` contain **79 files** (2019-06 to 2025-12 inclusive).

**Status**: Corrected in this revision.
